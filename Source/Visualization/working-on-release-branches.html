Working on Release Branches in Estream

WAKE UP! PAY ATTENTION!

Here are the rules for working on Ethane systems.

  1. Branches for Ethane Systems MUST be cut from the latest release branch, NEVER from develop.
  2. NEVER, EVER, merge the develop branch to the Ethane Systems release branch.
  3. Changes go through normal QA process through the "verify" and "validate" steps.
  4. Developer needs to get code changes approved by a SME.
  5. Developer sends an email to Sunil to merge the code.
  6. Someone with access to the release branch will merge the code and let the developer know once it is done.

Start a new feature:

Development MUST start from the [release] branch.

git flow feature start [my-feature] [release]

Work on your feature:

Work on your feature, and create one or more commits.

git add .
git commit -am "How did we miss that before?"

Finalize your work:

Get your feature through verification and validation.

Get a code review from the appropriate SME.

Merge your feature back to develop.

git checkout develop
git merge f-[my-feature]

DO NOT DELETE YOUR BRANCH!!!

Email Sunil to let him know your feature is finished. Be sure to include your branch name.

Sunil will finish your feature:

If your branch has not been merged to develop, WE WILL NOT MERGE IT TO ETHANE SYSTEMS.

This action:

  • Merges your feature into [release].
  • DOES NOT REMOVE THE FEATURE BRANCH.

git flow feature finish [my-feature]
git push

FINAL CLEANUP

  1. Sunil will email the release notes for the week's release.
  2. Said release notes will contain the branches that were merged to r-ethane-systems.
  3. If you branch was merged, you need to delete the branch.
